- fix: bump ts-cli to v0.92.0 — duplicate `column_id` → formula promotion (TML invariant I8/I5), shared by the from-Databricks and from-Snowflake Model builders (BL-132). When a source references one physical column both as a raw measure and as an aggregate metric (e.g. `F_TIME_TO_RESOLVE` + `AVG(TIMETORESOLVE__C)`), both classify as physical `columns[]` candidates with an identical `TABLE::col` — which ThoughtSpot rejects on import ("columns should have unique column_id values"). New shared helper `formula_common.promote_duplicate_column_ids` keeps the first occurrence as a `column_id` entry and re-expresses the rest as `fn ( [TABLE::col] )` aggregation formulas (SUM/AVERAGE/MIN/MAX/COUNT/MEDIAN/STDDEV/VARIANCE, plus I5's COUNT_DISTINCT → `unique count(...)`), wired into both `mv_build_model` and `sv_build_model` after `resolve_name_collisions`. A duplicate that is not a re-expressible aggregate (unmapped measure aggregation, or two bare dimensions on one column) is left in place so `ts tml lint` I8 still surfaces it. The bug affected both builders identically — the backlog's premise that from-Snowflake already handled it was inaccurate; the fix is new, factored to one home, and applied to both directions. Unit tests cover the helper's aggregation map + fail-loud paths and each builder's fact-col + aggregate-metric-on-same-column fixture (emitted TML passes I8 lint).
