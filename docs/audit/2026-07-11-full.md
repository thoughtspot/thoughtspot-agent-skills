# Repo Audit — full

**Scope:** full. **Method:** synthesis of raw findings across external angles 13 (product-currency,
per-platform), 14 (performance), 16 (dependency-currency), and internal angles 1 (dead files),
4 (tools quality), 5 (ts-cli gaps), 6 (testing value), 7 (PR validation), 11 (codification).
**Rubric:** `.claude/rules/repo-audit.md`. 46 raw findings deduplicated to **44**
(3 high / 16 medium / 25 low, including 4 positive/FYI records).

## Verdict

The repo is healthy at the gate level (angle 7 verified pre-commit/CI parity and closed bypass
routes), but three themes need action:

- 🔴 **Product currency bit Databricks again (1 high + companions)** — Metric View
  `parameters:` went GA (Runtime 18.2+), directly contradicting "runtime parameters don't
  exist in MVs" claims in five shipped files, and `ts databricks parse-mv` now rejects a
  supported GA construct as `unknown_key`. Snowflake's 2026-06-26 "SQL queries as logical
  tables" GA similarly obsoletes the sql_view "cannot be auto-mapped" verdict in both
  conversion directions.
- 🔴 **One real ts-cli crash + smoke-test drift (2 high)** — `resolve_profile` raises a raw
  TypeError on the current wrapped profiles-file format whenever `--profile`/`TS_PROFILE` is
  omitted (reproduced; caused by triplicated profile-parsing code with zero direct test
  coverage), and the ts-dependency-manager smoke test — the designated live-verification
  entry point for a destructive skill — never invokes the `ts dependency backup/apply-change/
  rollback` commands the skill now runs.
- 🟠 **The newest skill escaped every net (recurring blind spot)** — ts-convert-from-looker
  (shipped 2026-07-09) reintroduced the stdin JSON-array import boilerplate fixed elsewhere,
  is absent from the BL-100 codification program, has an undated smoke-test allowlist
  exemption, and dropped profile discovery from its authenticate step. Four findings, one
  root cause: nothing forces a new skill through the lessons already codified.
