# ts-audit unified HTML report — Design

**Status:** Approved (design) · **Date:** 2026-07-01 · **Author:** damian.waldron

Consolidate the current two-report output (audit findings + efficiency/object-reuse)
into a single, self-contained, shareable HTML file generated by a new `ts audit report`
CLI command. The report is a high-quality deliverable that SEs run against customer
environments and share directly with customers.

---

## 1. Purpose & scope

**In scope (v1):**

- A new `ts audit report` CLI command that reads the JSON output from `ts audit run`
  and produces a single self-contained HTML file.
- Extend `ts audit run` to include a `corpus` key with structural data (model metadata,
  table reuse, model overlaps, dependency counts, cluster URL).
- Five views: Dashboard, Model Scorecard, Cross-Model, Object Map, Cleanup.
- Persistent sidebar navigation, breadcrumb bar, severity filter, deep-link hash
  fragments, ThoughtSpot object links.
- ERD placeholder hook per model (ready for BL-058 integration).
- Delete `agents/cli/ts-audit/efficiency_report.py` (superseded).

**Out of scope:**

- **ERD renderer** (BL-058) — placeholder hook only; the shared renderer
  (`agents/shared/erd/`) is a separate deliverable.
- **Usage data / BI Server queries** — Phase 2, requires `ts data search` CLI command.
- **CoCo runtime** — CoCo cannot run Python; this is CLI-only.
- **PDF export** — users share the HTML file directly.

---

## 2. Audience

**Dual audience:** SEs run the audit, then share the HTML file with customers.

- **SE layer:** full technical detail — GUIDs, check IDs, metrics, thresholds,
  ThoughtSpot deep-links, ERD access.
- **Customer layer:** clean summary dashboard, plain-language findings, severity
  context, actionable recommendations.

Progressive disclosure bridges both: the dashboard is customer-readable; drill-down
into scorecards and cross-model views reveals SE-level detail.

---

## 3. Architecture — Approach B (template + data layer)

Two files in `ts_cli/audit/`:

| File | Responsibility |
|---|---|
| `report.py` | Read audit JSON, compute derived display data (heatmap cells, Sankey layout, name collision detection), inject payload into template, write HTML |
| `report_template.html` | Self-contained HTML file with CSS, JS, and a `<script id="audit-data">` placeholder. All rendering is client-side from the injected data. |

**Why not f-string HTML (the current pattern):** `efficiency_report.py` is 1,066 lines
of f-string HTML in Python. CSS and JS are buried in string literals with no editor
tooling. The template approach makes the HTML a real file — editable, previewable,
lintable — while the Python layer stays thin.

**Template injection:** `report.py` reads the template, replaces the `{{AUDIT_DATA}}`
marker inside the `<script id="audit-data">` tag with the serialised JSON payload.
The JavaScript reads `document.getElementById('audit-data').textContent` on
`DOMContentLoaded` and renders all views client-side.

## 4. Data pipeline

```
ts audit run --models ... --profile P    (produces enriched JSON to stdout)
        |
        v
ts audit report audit.json -o report.html   (pure renderer, no API calls)
        |
        v
   report.html  (self-contained, shareable)
```

Also supports piped input:
```bash
ts audit run --models ... --profile P | ts audit report -o report.html
```

### 4.1 Extended `ts audit run` output

Add a `corpus` top-level key alongside the existing `findings` and `summary`:

```jsonc
{
  "findings": [...],          // unchanged
  "summary": {...},           // unchanged
  "corpus": {
    "cluster_url": "https://myco.thoughtspot.cloud",
    "profile_name": "prod",
    "audit_date": "2026-07-01",
    "angles_run": ["A", "D", "H", "P", "S"],

    // Per-model structural metadata
    "models": [
      {
        "guid": "abc-123",
        "name": "Sales Model",
        "table_count": 12,
        "column_count": 87,
        "formula_count": 14,
        "join_count": 11,
        "join_depth": 4,
        "model_tables": [
          {"name": "DM_ORDERS", "fqn": "DB.SCH.DM_ORDERS"}
        ]
      }
    ],

    // Table reuse: which physical tables are shared across models
    "table_reuse": [
      {
        "fqn": "DB.SCH.DM_CUSTOMERS",
        "name": "DM_CUSTOMERS",
        "models": [
          {"name": "Sales Model", "guid": "abc-123"},
          {"name": "Purchasing", "guid": "def-456"}
        ]
      }
    ],

    // Model overlap pairs (from D7 analysis)
    "model_overlaps": [
      {
        "model_a": {"name": "Sales Model", "guid": "abc-123"},
        "model_b": {"name": "Revenue Model", "guid": "def-456"},
        "jaccard": 0.85,
        "shared_table_count": 8,
        "total_tables_a": 10,
        "total_tables_b": 9,
        "shared_tables": ["DM_ORDERS", "DM_CUSTOMERS"],
        "type": "high_overlap"  // identical | subset | high_overlap | conformed_reuse
      }
    ],

    // Dependents per model/table GUID
    "dependents": {
      "abc-123": [
        {"name": "Q1 Dashboard", "guid": "lb-001", "type": "LIVEBOARD"},
        {"name": "Sales Summary", "guid": "ans-001", "type": "ANSWER"}
      ]
    }
  }
}
```

The corpus is computed from the `AuditContext` that `build_context()` already
populates — models, tables, dependents, and metadata are all in memory. The
`cluster_url` is extracted from the profile config.

### 4.2 `ts audit report` command

```
ts audit report <input.json> [--output/-o <path>]
```

- Reads JSON from a file argument or stdin (piped).
- No `--profile`, no API calls — pure renderer.
- Writes self-contained HTML to the output path, or stdout if no `-o`.
- Exits non-zero if the input JSON is missing required keys (`findings`, `summary`).
- The `corpus` key is optional — without it, the Object Map and Cleanup views show
  "Run with the latest ts-cli to enable cross-model analysis."

## 5. Views & navigation

### 5.1 Navigation model

- **Persistent left sidebar** (~200px) — always visible. Lists all 5 views plus a
  collapsible model list. Active view highlighted. Badge counts for CRITICAL/HIGH
  next to Dashboard. Collapses to a hamburger icon on viewports under 768px.
- **Breadcrumb bar** at top — `Dashboard > Sales Model > D2`. Updates on every
  navigation. Each segment is clickable.
- **URL hash fragments** — every view and sub-view is deep-linkable:
  - `#dashboard`
  - `#model/{guid}`
  - `#model/{guid}/{angle}`
  - `#check/{check_id}`
  - `#object-map/{sub-tab}`
  - `#cleanup`
- **Severity filter bar** — persists across all views. Five toggle pills
  (CRITICAL, HIGH, MEDIUM, LOW, INFO). Toggling a severity hides/shows all
  findings of that severity across every view simultaneously. State stored in
  the hash or a JS variable (not localStorage — keeps the file stateless).

### 5.2 Object identity & display

Objects are identified internally by GUID. Display rules:

- **Primary label:** object name — always shown prominently.
- **Secondary label:** GUID — always shown in a smaller, muted style beneath or
  beside the name. Provides disambiguation naturally and is useful as a reference
  for SEs.
- **Name collisions:** when two objects share the same name, the GUID is already
  visible. No additional disambiguation logic needed — the dual-label pattern
  handles it by default.
- **ThoughtSpot link:** a small external-link icon beside the name. Opens the
  object in ThoughtSpot in a new tab. Omitted when `cluster_url` is empty.

### 5.3 ThoughtSpot deep-link URL patterns

| Object type | URL pattern |
|---|---|
| Model / Worksheet | `{cluster_url}/#/data/tables/{guid}` |
| Table | `{cluster_url}/#/data/tables/{guid}` |
| Answer | `{cluster_url}/#/saved-answer/{guid}` |
| Liveboard | `{cluster_url}/#/pinboard/{guid}` |

## 6. View details

### 6.1 Dashboard (landing page)

- **Header bar:** cluster URL (as a link), profile name, audit date, angles run.
- **Summary cards row:** one card per severity level showing the count. Cards are
  clickable — jumps to cross-model view filtered to that severity.
- **Cluster heatmap:** rows = models (sorted by worst severity first), columns = the
  5 angles. Each cell shows the worst severity for that model x angle pair,
  colour-coded. Interactions:
  - Click a cell → model scorecard filtered to that angle
  - Click a model name → model scorecard (all angles)
  - Click an angle column header → cross-model view filtered to that angle
- **Stats strip below heatmap:** total models, total tables, checks run, total findings.

### 6.2 Model Scorecard

- **Model header:** name (primary), GUID (secondary/muted), table count, column
  count, formula count, join depth. ThoughtSpot link icon. ERD button (placeholder —
  see section 7).
- **Angle sections:** one collapsible section per angle that was run (A, D, H, P, S).
  Each shows:
  - Angle summary: worst severity + finding count
  - Finding cards: check ID, severity badge, detail text, metric + threshold where
    present (e.g. "25% — threshold: 80%"), recommendation text with follow-up skill
    link
  - CRITICAL/HIGH expanded by default; MEDIUM/LOW/INFO collapsed
- **Dependents panel** (collapsible at bottom): lists answers, liveboards, and sets
  that consume this model. Each row shows name, type badge, GUID, and TS deep-link.

**Recommendation mapping for finding cards:**

| Check pattern | Recommendation |
|---|---|
| A1-A5 | `/ts-object-model-coach` to improve descriptions, synonyms, AI context |
| D7 (model overlap) | `/ts-dependency-manager` (Repoint mode) to consolidate |
| D8 (duplicate tables) | `/ts-dependency-manager` (Remove mode) |
| H3 (unnecessary hidden columns) | Remove from model via TML reimport |
| H4 (orphan models) | `/ts-dependency-manager` (Remove mode) if confirmed unused |
| H8 (formula promotion) | `/ts-object-answer-promote` |
| S1-S5 (security) | Manual review via ThoughtSpot UI or TML reimport |

### 6.3 Cross-Model

- **Filterable table:** one row per finding across all models. Columns: severity
  badge, check ID, angle, model name (linked to scorecard), object name, object
  type, detail.
- **Sort on any column** — click header to toggle ascending/descending.
- **Text filter** — searches across model name, object name, and detail.
- **Severity filter** — same global filter bar, applied here.
- **Group-by toggle:** flat list (default) or grouped by check ID (shows all D2
  findings together with a count header).

### 6.4 Object Map

Four sub-tabs within this view:

| Sub-tab | Content |
|---|---|
| **Overlaps** | Model pairs with overlap classification (identical, subset, high_overlap, conformed_reuse). Each row shows both model names + GUIDs, Jaccard score, shared table count, severity. A "Compare" expandable shows the shared table list. |
| **Duplicates** | Groups of table objects pointing to the same physical table. Each group shows the physical FQN, the duplicate TS table objects, and which models reference each. |
| **Table Reuse** | Sankey diagram (physical tables on left → models on right) for shared tables. Below it, a sortable table: FQN, table name, model count, model names (linked). |
| **Dependencies** | Two horizontal bar charts: tables-per-model (blue) and models-per-table (purple, shared tables only). Sortable detail tables beneath each chart. Model/table names link to TS. |

All sub-tabs render their charts client-side from the corpus data (SVG built by
JavaScript). No pre-rendered SVG in the template.

### 6.5 Cleanup

- **Orphan models (H4):** table with name, GUID, TS link, dependent count (zero),
  recommendation.
- **Orphan sets (H5):** same format — name, GUID, TS link, consuming answer/liveboard
  count (zero).
- **Stale objects (H10):** split into model-level and column-level sections. Each row
  shows the matched stale pattern, the object name, the model it belongs to, and the
  GUID.
- **Checkbox column** (client-side only, not persisted) on each row so the user can
  build a "cleanup list" and copy the selected items.

## 7. ERD hook

Each model scorecard has a "View ERD" button in the model header. The template
reserves:

- A `<div id="erd-{guid}" class="erd-container">` placeholder per model.
- A `data-erd-ready="false"` attribute on the button.
- A JavaScript hook: `window.renderERD(containerId, modelData)` — when BL-058 ships,
  the shared renderer populates this function.

Until BL-058 is wired in, clicking the button shows a styled message:
"ERD diagram — coming soon. Use `/ts-object-model-erd` for standalone ERD rendering."

**Integration path (BL-058 follow-up):**

1. Inline `renderer.css` and `renderer.js` from `agents/shared/erd/` into the template.
2. `report.py` populates the MODEL schema (ERD design spec §3) from the corpus data
   for each model.
3. Set `data-erd-ready="true"` — the button activates, clicking renders the ERD
   inline in the container.
4. No template restructuring needed — the hook is designed for drop-in.

## 8. Size optimisation

Target: **under 1MB** for a 100-model audit. Comfortable for email and Slack sharing.

### 8.1 Strategies

1. **Client-side rendering** — SVG charts (Sankey, bar charts, heatmap cells) are
   built by JavaScript from the data payload. Zero static SVG cost in the template.
2. **Compact data serialisation** — `report.py` transforms the verbose `ts audit run`
   JSON into a compact payload with short keys (`n` for name, `g` for guid, `s` for
   severity, etc.) before injecting into the template. The `ts audit run` output
   format (section 4.1) is unchanged — compaction is a rendering concern only.
   A key legend in the JavaScript maps short keys back to readable names. Typically
   40-60% smaller than verbose keys.
3. **CSS minification** — the template source (`report_template.html`) uses readable,
   commented CSS for development. `report.py` strips whitespace and comments from the
   CSS at injection time, so the output HTML is minified without losing source readability.
4. **Deferred DOM construction** — views are built on first navigation, not all at
   page load. A 100-model audit does not construct 100 scorecard DOMs upfront.
5. **Data deduplication** — model names and GUIDs go in a lookup array, referenced
   by index elsewhere in the payload. Findings reference the model index, not
   repeated name+GUID strings.
6. **No external dependencies** — no web fonts, no icon library. Severity colours
   via CSS custom properties. Simple Unicode glyphs for icons.

### 8.2 Size budget

| Scenario | Template | Data payload | Total |
|---|---|---|---|
| 5 models, 50 findings | ~60KB | ~30KB | ~90KB |
| 20 models, 200 findings | ~60KB | ~120KB | ~180KB |
| 100 models, 1000 findings | ~60KB | ~500KB | ~560KB |

## 9. Visual design

The `artifact-design` skill will be used during implementation to produce the CSS
and visual identity. Design constraints for that process:

- **Severity colour system** as CSS custom properties — used consistently across
  heatmap cells, finding cards, filter pills, severity badges, and chart elements.
  Semantic colour (good/warning/critical) is separate from the accent palette.
- **Typography:** system font stack (`system-ui, -apple-system, "Segoe UI", Roboto,
  sans-serif`). No web fonts (CSP and size constraints). Monospace for GUIDs, check
  IDs, FQNs (`ui-monospace, "SF Mono", Menlo, Consolas, monospace`).
- **Information density:** the report is scanned and operated, not read linearly.
  Surface summaries before detail. Encode state in form (pills, stripes, badges)
  so what needs attention reads at a glance.
- **Print:** `@media print` hides sidebar and filter bar, expands all collapsed
  sections, uses black-and-white severity labels instead of colour-only encoding.
- **Consistency with ERD mockup:** adopt the same CSS custom property names and
  colour values from the validated ERD mockup (`--ground`, `--surface`, `--ink`,
  `--muted`, `--accent`, `--crit`, `--warn`, `--ok`, etc.) so the ERD integrates
  visually when BL-058 ships.

## 10. Testing strategy

Testing is primarily through **Artifact previews in the browser** during
implementation.

### 10.1 Fixture data generator

A Python function (`generate_test_data()`) that produces realistic audit JSON with
configurable parameters:

- `model_count` — number of models (default 5)
- `findings_per_model` — average findings density (default 10)
- `include_corpus` — whether to include corpus data (default True)
- `name_collisions` — number of models with duplicate names (default 0)
- `empty_angles` — angles with zero findings (default none)
- `cluster_url` — TS base URL (default "https://demo.thoughtspot.cloud")

Edge cases built in: zero findings for some models, models with 100+ columns,
a model with only INFO findings, missing GUIDs on some dependents.

### 10.2 Visual test matrix

Each implementation task ends with a browser preview (via the Artifact tool)
verifying:

| Category | What to verify |
|---|---|
| **Navigation** | All 5 views reachable via sidebar; active state highlights correctly; breadcrumb updates on every navigation; back/forward browser buttons work |
| **Collapse/expand** | Angle sections toggle smoothly; nested collapses (angle > finding card) work independently; expand-all / collapse-all if present |
| **Resize** | Sidebar collapses to hamburger under 768px; heatmap scrolls horizontally when many models; tables get horizontal scroll on narrow viewports; no body-level horizontal scroll |
| **Severity filter** | Toggling a pill hides/shows findings across all views simultaneously; filter state persists across view navigation; "0 findings match" message when all filtered out |
| **Deep-links** | Hash navigation works on fresh page load; copying URL and opening in new tab lands on the right view; model GUIDs in URLs resolve correctly |
| **TS links** | External-link icons present on all object names with GUIDs; correct URL pattern per object type; open in new tab; omitted gracefully when cluster_url is empty |
| **Sort/filter** | Cross-model table sorts on each column; text filter searches model name, object name, detail; sort direction toggles on repeated click |
| **Edge cases** | Zero findings for an angle (section shows "No findings"); single model (heatmap is one row); 50+ models (heatmap scrolls, sidebar model list scrolls); name collisions (GUIDs disambiguate); empty corpus (Object Map/Cleanup show graceful message) |
| **Object Map** | Sankey renders with correct flow widths; bar charts scale to data; sub-tab navigation works; overlap "Compare" expand shows shared table list |
| **Cleanup** | Checkboxes toggle independently; "copy selected" copies name+GUID pairs; orphan/stale sections show correct finding data |

### 10.3 Structural assertions (pytest)

Lightweight tests on `report.py`:

- Template placeholder `{{AUDIT_DATA}}` is replaced in output.
- Output contains no external resource references (`http://` or `https://` in
  `src` or `href` attributes, excluding TS deep-links and the cluster_url header link).
- Injected data payload parses as valid JSON.
- Output file size is under 1MB for the 100-model test fixture.
- Missing `corpus` key produces a valid report (Object Map/Cleanup show fallback).
- Empty `findings` list produces a valid report with "No findings" messaging.
